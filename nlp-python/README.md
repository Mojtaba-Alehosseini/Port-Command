# Port Command Genova — NLP side

Two local Python services back the game's natural-language layer:

| Service | Port | Python | venv | Task |
|---|---|---|---|---|
| **Rasa OSS NLU** (intent + entity) | 5005 | **3.10** | `.venv-rasa` | 12 (this) |
| **Flask LLM sidecar** (explanations) | 5006 | 3.11 | `.venv` | 13 |
| OWL → assets converter (`ontology_to_assets.py`) | — | 3.11 | `.venv` | 02 |

The Java/JADE side talks to both over HTTP/JSON (bridge wiring is task 14).

## Why two venvs

`rasa 3.6.21` (the final Rasa **OSS** release) declares `Requires-Python >=3.8,<3.11`,
so it **cannot** share an interpreter with the 3.11 tooling. The Rasa service therefore
lives in its own Python 3.10 venv (`.venv-rasa`); everything else stays on 3.11 (`.venv`).
They are separate processes, so the split costs nothing at runtime.

## Setup

```bash
cd nlp-python

# 3.11 venv — converter, LLM sidecar (task 13), and ALL pytest
py -3.11 -m venv .venv
.venv/Scripts/pip install -r requirements-llm.txt

# 3.10 venv — Rasa NLU service only  (installs TensorFlow 2.12, spaCy, sklearn ...)
py -3.10 -m venv .venv-rasa
.venv-rasa/Scripts/pip install -r requirements-rasa.txt
.venv-rasa/Scripts/rasa --version          # -> Rasa Version 3.6.21, Python 3.10.x
```

If Python 3.10 is not installed: `winget install Python.Python.3.10`.

## The training corpus (authoring-CSV workflow)

`rasa/data/nlu_authoring.csv` is the **single source of truth** for the corpus
(columns: `intent, text, annotated_text, split`). Edit it in a spreadsheet, then
regenerate the Rasa YAML — never hand-edit the generated files:

```bash
.venv/Scripts/python csv_to_nlu.py          # CSV -> rasa/data/nlu.yml + rasa/tests/test_holdout.yml
.venv/Scripts/python csv_to_nlu.py --check  # CI: fail if the on-disk YAML is stale
```

`csv_to_nlu.py` **fails loudly** if any normalised utterance appears in both the
`train` and `test` splits (leakage), is duplicated within a split, or carries an
unknown intent — keeping the held-out set honest. Covered by `test_csv_to_nlu.py`.

Corpus size (PROJECT_DEFINITION § 6.3 v1.1): **9 intents × (50 train + 10 holdout) =
540 utterances** (450 train, 90 holdout), ≈ 34 standard / 8 informal / 5 typo /
3 fragment-vocative per intent in the training split.

## Train

```bash
./train_rasa.sh        # regenerates NLU data from the CSV, then `rasa train` (< 5 min CPU)
```

The model is written to `rasa/models/portcmd_nlu.tar.gz` (gitignored).

## Test — the acceptance gate

```bash
./test_rasa.sh
```

- **Gate (the ONLY pass/fail):** intent weighted-F1 **≥ 0.85** on the 90-utterance
  holdout (`rasa/results/holdout/intent_report.json`).
- **Reported, NOT gated:** entity (NER) weighted-F1 from the same run
  (`rasa/results/holdout/DIETClassifier_report.json`).
- **In-dev noise floor:** 5-fold cross-validation over `nlu.yml` (informational).

Confusion matrices and per-class reports land under `rasa/results/` (gitignored).

## Intents & entities (PROJECT_DEFINITION § 6.3 — do not add/rename)

**9 intents:** `propose_offer`, `counter_offer`, `accept_deal`, `reject_deal`,
`query_status`, `set_constraint`, `set_policy`, `request_help`, `cancel_action`.

**8 entities:** `vessel_name`, `vessel_class`, `berth_id`, `tug_name`,
`time_expression`, `price_expression`, `cargo_class`, `hazmat_code`.

Entity **values** are normalised to the ontology vocabulary
(`rasa/data/ontology_vocab.yml`, generated by task 02) — e.g. `container ship`
→ `container_vessel`, `berth 3` → `berth_3`, `class 3` → `hazmat_class_3` — via
`EntitySynonymMapper`. `tug_name` values are the AgentRoster ids `tug_1..tug_4`.

## LLM sidecar (task 13)

A Flask service on **port 5006** that turns a decision trace into a 2-3 sentence
prose explanation for the Assistant. It loads one local instruction model once at
startup and serves deterministic (`do_sample=False`, greedy) completions.

```bash
cd nlp-python
./start_llm.sh          # foreground; Windows: start_llm.bat
```

`/health` returns **503 `{"ready": false}`** while the model loads and **200
`{"ready": true, ...}`** once warmup finishes. `POST /explain` accepts:

```jsonc
{
  "prompt": "Recommend accept at 2000 ...",   // required (plain text; the sidecar
                                               //   applies the model's OWN chat template)
  "system": "You are the harbour-master's ...",// optional system instruction
  "required_numbers": ["2000", "3"],           // optional; validator inputs
  "required_entities": ["Aurora"],             // optional; validator inputs
  "validate": true                             // optional; run the validator
}
// -> {"text": "...", "validated": true|false|null, "gen_seconds": 1.23}
```

`validated` is `null` when `validate` was not requested; `false` never blocks the
response — the Java caller decides whether to use the text or fall back to the
template (planning/13 §13.4). Prompts over `LLM_MAX_PROMPT_CHARS` (4096) → 400;
model-not-ready → 503; generation error → 500 (type only, no stack trace).

### Model selection & quantization

| `settings.json` `llm.model` / `LLM_MODEL` | HF repo | License | Notes |
|---|---|---|---|
| `phi-4-mini` (default) | `microsoft/Phi-4-mini-instruct` | **MIT** | causal LM, `trust_remote_code=True` |
| `gemma-3-4b` | `google/gemma-3-4b-it` | **Gemma Terms** | **🔒 HF-gated** — see below; multimodal checkpoint |

`settings.json` is owned by task 15 and is **tolerated absent** — without it the
sidecar defaults to `phi-4-mini`. The env var `LLM_MODEL` overrides it.

**Gemma is HF-gated.** To use `gemma-3-4b` you must accept the license at
<https://huggingface.co/google/gemma-3-4b-it> and provide a token
(`huggingface-cli login`). The code branch (multimodal `Gemma3ForConditionalGeneration`
+ `AutoProcessor`) is implemented and unit-tested for config/branch resolution, but
the real Gemma download is **deferred to the report-time A/B comparison** — CI never
loads it.

### Environment variables

| Var | Default | Meaning |
|---|---|---|
| `LLM_MODEL` | *(settings.json → `phi-4-mini`)* | `phi-4-mini` \| `gemma-3-4b` |
| `LLM_DEVICE` | `cpu` | `cpu` \| `cuda` |
| `LLM_QUANT` | `none` | `none` (fp16/bf16) \| `int4` \| `gguf` \| `onnx` (fast CPU path — task 13b) |
| `LLM_HOST` / `LLM_PORT` | `127.0.0.1` / `5006` | bind address |
| `LLM_MAX_NEW_TOKENS` | `120` | generation cap (80 also fits 2-3 sentences, faster) |
| `LLM_MAX_PROMPT_CHARS` | `4096` | reject longer prompts with 400 |
| `LLM_LOG_LEVEL` / `LLM_LOG_FILE` | `INFO` / *(console)* | logging |
| `LLM_AUTOLOAD` | `1` | `0` disables the import-time loader (used by tests) |
| `LLM_VOCAB_PATH` | `rasa/data/ontology_vocab.yml` | validator gazetteer source |
| `LLM_ONNX_DIR` | `~/.cache/port-command-genova/…/cpu-int4-rtn-block-32-acc-level-4` | local ONNX-INT4 model dir (task 13b) |

### Fast CPU backend: ONNX Runtime GenAI INT4 (`LLM_QUANT=onnx`) — recommended

**The fast CPU path** (task 13b). Phi-4-mini int4 via ONNX Runtime GenAI measured
**~4.56 tok/s ≈ 17 s/explanation** on the dev laptop (i7-8850H, AVX2, no AVX-512) —
**~31× faster** than the fp16 default's ~9 min (see Performance). Deterministic
(byte-identical), all figures preserved. Prebuilt CPU wheel (in `requirements-llm.txt`):
no CUDA, no compile.

One-time setup — weights live **outside the repo** (~4.9 GB), never committed:

```bash
huggingface-cli download microsoft/Phi-4-mini-instruct-onnx \
  --include "cpu_and_mobile/cpu-int4-rtn-block-32-acc-level-4/*" \
  --local-dir "$HOME/.cache/port-command-genova/Phi-4-mini-instruct-onnx"
```

Then `start_llm.{sh,bat}` **auto-selects** `LLM_QUANT=onnx` when the model is present
(the portable default stays `none`); or set `LLM_QUANT=onnx` explicitly. `LLM_ONNX_DIR`
overrides the model location. onnx is **Phi-4-only** (the ONNX repo hosts only Phi-4);
the sidecar refuses `LLM_QUANT=onnx` for `gemma-3-4b`.

### Quantized fallbacks (low-RAM machines)

Default `LLM_QUANT=none` loads fp16/bf16 (~7.6 GB RAM). For a smaller/faster
footprint, install the optional extra (commented in `requirements-llm.txt`) and set:

- **`LLM_QUANT=gguf`** (CPU-quantized, ~2-3 GB; **superseded by `onnx` above** as the
  fast CPU path): loads `unsloth/Phi-4-mini-instruct-GGUF` via `llama-cpp-python`.
  **Caveat (measured):** the prebuilt wheels don't work on the dev CPU — see
  **Performance** below; a from-source AVX2 build is required there.
- **`LLM_QUANT=int4`** (**CUDA only** — bitsandbytes 4-bit kernels need a GPU):
  `pip install bitsandbytes`, loads `unsloth/Phi-4-mini-instruct-bnb-4bit` (~3 GB).

### Performance (measured 2026-07-12, tasks 13 + 13b)

> ✅ **Fast path — `LLM_QUANT=onnx` (ONNX Runtime GenAI CPU-INT4, task 13b):** on the dev
> laptop (i7-8850H, AVX2, no AVX-512) — **load ~18 s, ~17 s/explanation, ~4.56 tok/s**,
> deterministic (byte-identical), all figures preserved — **~31× faster** than fp16. The
> recommended backend; see the fast-backend section above to enable it.

> ⚠️ **On CPU, fp16 is correct but far too slow for interactive play.** On the dev
> laptop (i7-8850H, no AVX-512), Phi-4-mini at `LLM_QUANT=none` measured **load 37 s,
> generation ≈ 9 minutes** for a ~66-token explanation (0.1 tok/s — the CPU has no
> native bf16, so PyTorch emulates it ~100× slower). Output is correct and
> deterministic — just slow.

On the fp16 backend the Assistant treats the LLM as **optional polish**: the Java
`LLMBridge` (task 14) calls `/explain` asynchronously with a config-driven timeout
and shows the **template text** whenever the LLM does not answer in time (always at
fp16). **With `LLM_QUANT=onnx` (task 13b) the LLM answers in ~17 s and the player sees
real LLM prose.** The other CPU/GPU paths that were considered:

- **GGUF** would be the CPU answer, but is **blocked on this CPU by a wheel
  version-squeeze**: `llama-cpp-python` 0.3.x (which supports the Phi-4 GGUF arch)
  ships an AVX-512 wheel that crashes (`0xc000001d`) on Coffee Lake, and 0.2.90 (AVX2)
  is too old to load the Phi-4 GGUF. A working build needs compiling from source:
  `CMAKE_ARGS="-DGGML_AVX512=OFF -DGGML_AVX2=ON -DGGML_NATIVE=OFF" pip install llama-cpp-python`.
- **GPU** (the P2000, 5 GB) fits INT4 (~3 GB) but not fp16 (~7.6 GB); INT4 via
  bitsandbytes is untested on Pascal + Windows.

### Weights cache

`transformers` downloads to the HuggingFace hub cache (`~/.cache/huggingface/hub`,
or `%USERPROFILE%\.cache\huggingface\hub` on Windows) — **outside the repo**, so
nothing is committed. First run downloads ~7.6 GB; pre-fetch with
`huggingface-cli download microsoft/Phi-4-mini-instruct`.

### The hallucination validator

`validator.py` mirrors the task-10 Java `HallucinationValidator` (planning/10
§10.4): (1) every required number must appear in the output — with v1.1 numeric
normalisation so `€2,000` / `2 000` / `85%` compare equal to `2000` / `2000` /
`85`; (2) every capitalised, non-sentence-initial, non-common-word token must be
in the gazetteer (ontology entities ∪ per-call `required_entities`) — catching
fabricated vessel/tug names. The Java side additionally runs a positive-control
(`allFigures()`); that needs the full `Recommendation` object the sidecar never
receives, so it stays Java-side.

### Manual real-model smoke

```bash
./start_llm.sh &                                             # loads the model
.venv/Scripts/python -m llm_sidecar.tests.smoke_real_model   # health, determinism, validator, latency
```

Not part of the commit gate — it loads real weights. The mocked pytest suite
(`llm_sidecar/tests/`) never loads weights and runs in <1 s.

## Notes

- **`set_policy` is a routing label only.** The Java layer hands the raw player
  text to `PolicyParser` (task 10, a regex-DSL) and ignores Rasa's entities on
  that path. The second Rasa pipeline was cut (PROJECT_DEFINITION § 9 item 6).
- **No Duckling.** Numeric/price/time normalisation is done Java-side
  (`PreprocessRegex`, task 14); Rasa only tags the spans.
- **Confidence thresholds** (used Java-side): 0.60 high / 0.40 medium / < 0.40 low.
